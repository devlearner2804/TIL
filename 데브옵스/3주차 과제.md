## Git 작업 흐름

<details>
<summary>1. Branch 생성</summary>

브랜치(Branch)는 기존 코드에 영향을 주지 않고 새로운 기능을 개발하거나 수정 작업을 진행하기 위해 사용하는 독립적인 작업 공간이다.  
하나의 프로젝트에서 여러 사람이 동시에 작업할 수 있도록 도와주며, 작업이 완료되기 전까지는 메인 코드에 영향을 주지 않는다.

새로운 브랜치를 생성하면 해당 브랜치에서 자유롭게 코드를 수정할 수 있고, 이후 메인 브랜치와 병합(Merge)할 수 있다.

</details>

<details>
<summary>2. push 후 PR 생성</summary>

브랜치에서 작업이 완료되면 변경된 내용을 로컬 저장소에 커밋(commit)한 후, 원격 저장소(예: GitHub)에 업로드(push)한다.

이후 Pull Request(PR)를 생성하는데, 이는  
“내가 작업한 내용을 메인 브랜치에 합쳐도 되는지 검토해달라”는 요청이다.

PR을 통해 코드 리뷰를 진행할 수 있고, 다른 개발자가 코드를 확인한 뒤 문제가 없으면 병합을 승인한다.

</details>

<details>
<summary>3. merge</summary>

Merge는 브랜치에서 작업한 내용을 메인 브랜치에 합치는 과정이다.

Pull Request가 승인되면 merge를 통해 코드가 통합되며,  
이로써 해당 기능이나 수정 사항이 프로젝트에 반영된다.

이 과정에서 코드 충돌(Conflict)이 발생할 수 있으며,  
이 경우 개발자가 직접 충돌을 해결한 후 다시 merge를 진행해야 한다.

</details>
## 네트워크 핵심 개념

<details>
<summary>1. IPv4, IPv6</summary>

IPv4는 인터넷에서 사용되는 대표적인 IP 주소 체계로, 32비트 주소를 사용하며 약 43억 개의 주소를 표현할 수 있다.

하지만 인터넷 사용자 증가로 주소가 부족해지면서 IPv6가 등장하였다.

IPv6는 128비트 주소를 사용하여 거의 무한에 가까운 주소를 제공하며, 보안성과 효율성이 향상되었다.

차이점

- IPv4: 32비트, 점으로 구분 (예: 192.168.0.1)
- IPv6: 128비트, 콜론으로 구분 (예: 2001:db8::1)

</details>

<details>
<summary>2. ARP / RARP</summary>

ARP는 IP 주소를 이용하여 해당 장치의 MAC 주소를 알아내는 프로토콜이다.

즉, “이 IP 가진 장치 누구야?”라고 물어보는 과정이다.

RARP는 반대로 MAC 주소를 통해 IP 주소를 알아내는 방식이다.

현재는 DHCP에 의해 거의 사용되지 않는다.

</details>

<details>
<summary>3. 3-way Handshake</summary>

TCP 통신에서 연결을 설정하기 위한 과정이다.

클라이언트와 서버가 서로 연결 준비가 되었는지 확인한다.

과정

1. 클라이언트 → 서버 : SYN (연결 요청)
2. 서버 → 클라이언트 : SYN + ACK (요청 수락)
3. 클라이언트 → 서버 : ACK (확인)

이 과정을 통해 신뢰성 있는 연결이 생성된다.

</details>

<details>
<summary>4. 4-way Handshake</summary>

TCP 연결을 종료할 때 사용하는 과정이다.

과정

1. 클라이언트 → 서버 : FIN (종료 요청)
2. 서버 → 클라이언트 : ACK (확인)
3. 서버 → 클라이언트 : FIN (종료 준비)
4. 클라이언트 → 서버 : ACK (최종 확인)

</details>

<details>
<summary>3-way vs 4-way Handshake 비교</summary>

- 3-way: 연결 “시작” 과정
- 4-way: 연결 “종료” 과정
- 3-way는 3단계, 4-way는 4단계
- 4-way는 데이터 송수신 종료 확인이 필요해서 단계가 더 많음

</details>

<details>
<summary>5. 브로드캐스트, 유니캐스트, 멀티캐스트, 애니캐스트</summary>

유니캐스트는 1:1 통신 방식이다. (한 장치 → 한 장치)

브로드캐스트는 1:전체 통신으로, 같은 네트워크의 모든 장치에게 데이터를 전송한다.

멀티캐스트는 1:그룹 통신으로, 특정 그룹에 속한 장치들에게만 데이터를 전송한다.

애니캐스트는 여러 서버 중 가장 가까운 하나의 서버로 데이터를 보내는 방식이다.

</details>

<details>
<summary>6. TCP / UDP</summary>

TCP는 연결형 프로토콜로, 데이터의 순서와 신뢰성을 보장한다.  
속도는 상대적으로 느리지만 안정적인 통신이 가능하다.

UDP는 비연결형 프로토콜로, 속도가 빠르지만 신뢰성이 낮다.  
데이터 손실이 발생할 수 있다.

</details>

<details>
<summary>7. OSI 7 Layer</summary>

네트워크 통신 과정을 7단계로 나눈 모델이다.

1. 물리 계층 (Physical) – 실제 전기 신호 전송  
2. 데이터링크 계층 (Data Link) – MAC 주소, 오류 검출  
3. 네트워크 계층 (Network) – IP 주소, 경로 설정  
4. 전송 계층 (Transport) – TCP/UDP  
5. 세션 계층 (Session) – 연결 유지 및 관리  
6. 표현 계층 (Presentation) – 데이터 형식 변환, 암호화  
7. 응용 계층 (Application) – 사용자와 직접 상호작용  

</details>

<details>
<summary>8. 브리지 (Bridge)</summary>

브리지는 네트워크를 분할하여 트래픽을 줄여주는 장비이다.

MAC 주소를 기반으로 데이터를 전달하며, 충돌 도메인을 나누는 역할을 한다.

</details>

<details>
<summary>9. 스위치 (Switch)</summary>

스위치는 브리지의 발전된 형태로, 여러 장치를 연결하는 네트워크 장비이다.

MAC 주소를 기반으로 데이터를 필요한 장치에만 전달하여 효율적인 통신이 가능하다.

브리지보다 속도가 빠르고 포트 수가 많아 현대 네트워크에서 주로 사용된다.

</details>
## 리눅스 /etc 및 네트워크 개념

<details>
<summary>1. DNS 동작 흐름</summary>

도메인을 IP 주소로 바꾸는 과정

1. 사용자가 도메인 입력  
2. /etc/hosts 파일 먼저 확인  
3. 없으면 DNS 서버에 요청  
4. IP 주소를 받아서 해당 서버에 접속  

</details>

<details>
<summary>hosts vs DNS 차이</summary>

## 핵심 차이 정리

| 구분 | hosts | DNS |
| --- | --- | --- |
| 적용 범위 | 내 컴퓨터 | 전 세계 |
| 방식 | 수동 설정 | 서버 자동 |
| 속도 | 빠름 | 상대적으로 느림 |
| 우선순위 | 높음 | 낮음 |

</details>

<details>
<summary>네트워크 설정 파일</summary>

- /etc/resolv.conf : DNS 서버 설정  
- /etc/hosts : 도메인-IP 매핑  
- /etc/netplan/ : IP, 게이트웨이 등 네트워크 전체 설정  

</details>

<details>
<summary>UID / GID 개념</summary>

- UID : 사용자 번호  
- GID : 그룹 번호  

컴퓨터는 사용자 이름이 아니라 번호로 사용자를 구분한다.

</details>

<details>
<summary>root 계정 특징</summary>

- UID = 0 (관리자)  
- 모든 파일 접근 가능  
- 모든 명령 실행 가능  
- 실수 시 시스템 전체에 큰 영향을 줄 수 있음  

</details>
